# Agent-Native Platform Plan

**Date:** 2026-07-06 (rewritten — supersedes the prior draft of this file)
**Status:** Proposed masterplan. Supersedes the *agent-experience* threads of PAVED-ROAD / SMARTER-AGENTS / AGENT-BUILD-LOOP for prioritization, and **unifies** the *substrate* threads of LIVING-APPS, CONVERSATION-STATE-MACHINE, AGENT-PRIMARY-POSTURE, DURABLE-TASK-SPINE, AGENTIC-SYSTEMS-ARCHITECTURE, SWIFT and COGNITIVE-LOOPING into one primitive set (§8 reconciliation).
**Thesis:** Agentis was built so agents could *10x their work inside it*. Today it does the opposite — even frontier agents duplicate resources, cannot wire primitives, cannot make a persistent agent exist, and re-derive semantics by trial and error. **The cause is not missing features. It is that Agentis already contains ~80% of every primitive an agent-native platform needs — persistent agents, per-subject durable entities, owned connections, live graph evolution, outcome learning, a living console — but each is built as a *separate, narrowly-scoped, half-wired, unadvertised* subsystem, so no agent (and no human) can see or drive the whole.** This plan does not add a ninth subsystem. It **unifies the eight into six legible primitives**, generalizes the narrow ones, wires the inert ones, enforces the unenforced ones, and makes the result something an agent can actually reason about. Every fix is general-purpose — no use-case patches (`feedback_agentis_general_not_use_case`).

Derived from a six-front codebase audit (persistent-agent lifecycle, connections/permissions, execution/durable-state, apps/interface/experimentation, tool surface, guidance) cross-checked against 2025–2026 external research on durable execution, actor models, agent authorization, code-mode, and agent evaluation.

---

## Part 0 — The reframe: fragmentation, not absence

The operator's stress test — *"a persistent agent that every 5 min discovers leads, sends a deterministic token-free first message from my connection, waits (days, out of order) for each reply, has a small model write message 2 and classify it, and on a positive reply triggers a build-and-deliver, then stops — with A/B testing of the messages and connections owned by specific agents"* — reads like a list of missing features. It is not. **Every one of those capabilities already has a half-built home in the tree.** The reason the build failed, and the reason agents get lost, is that those homes are disconnected:

| Concern | Parallel subsystems that each own a slice | The problem |
|---|---|---|
| **Cognition** | `agents` row (identity) · `AgentSession` (run-scoped) · `ChatSession` (chat-scoped) | Nothing thinks *across* runs; the mind dies with the run. |
| **Per-subject state** | `WorkflowRunState` (per run) · `ConversationContactState` (per contact) · `app_contacts` (per relationship) | Three "per-something" stores; a lead's life is smeared across all three. |
| **Scheduling / wake** | `appOrchestrator` cron (wakes *workflows*) · `CommandHeartbeat` (wakes *manager agents only*) · `ProactiveFollowupService` (wakes *contacts*, canned text) · `ListenerRuntime` (wakes *workflows* on events) | Four schedulers, none general; a plain agent cannot be scheduled at all. |
| **Connections** | `channel_connections.agentId` (inbound routing) · `credentialVault` (workspace secrets) · `groundingAgentGrants` (the *real* per-agent ACL — for web search) | The correct ACL shape exists, bound to the wrong resource; sending is unchecked. |
| **Authoring** | workflow graph JSON · conversation-script JSON · listener config | Three DSLs glued by opaque IDs; agent-first orchestration specced but unbuilt. |
| **Measurement** | `runAnalytics` (run success rate) · `appLearning` (qualitative lessons) · `SWIFT` verdict (accepted/not) | Real graders and analytics exist; no experiment/variant/metric substrate joins them. |
| **Legibility** | `orchestratorPrompt` (withholds the ontology from MCP agents) · `routes/mcp.ts` (flattens errors) · ~90 plan docs | The platform cannot describe itself; its names promise things they don't deliver. |

Prior masterplans each added *one more spine* (LIVING-APPS added the resident-agent spine; CONVERSATION-STATE-MACHINE added the script spine; AGENT-PRIMARY-POSTURE proposed the agent-owns-the-loop spine). Each was individually good and individually shipped or specced. **Adding spines is the disease, not the cure.** The 10x is to collapse them.

## Part 1 — The seven fractures (root cause, grounded in code)

Each fracture is verified at file:line, names the subsystems it splits, and points at the unification.

### F1 — Cognition has no persistent home
An "agent" is a row in `agents` (a harness *connection descriptor*, not a mind). The only thing that thinks — `AgentSession` — is keyed `(runId, nodeId)` (`agentSession.ts:148-156`) and **dies with its run**; its own docstring admits "between LLM inference calls the session is just this row" (`agentSession.ts:1-14`). Identity (`agentIdentity.ts`) and memory (`agentMemory.ts`, `scopeId=agentId`) persist — but *nothing acts on them across runs*. The single mechanism that wakes an agent unprompted and lets it act, `CommandHeartbeat` (`commandHeartbeat.ts:41-109`), is **hardcoded to orchestrator/domain-manager roles** (`isOrchestratorRole`, `commandScope.ts:34-36`) and re-enters the agent with **empty history** (`ChatSessionExecutor.turn([], …)`, `bootstrap.ts:1452`) — no working-memory continuity. → *A persistent agent cannot exist for the 99% of agents that aren't managers, and even managers wake amnesiac.*

### F2 — The per-subject durable entity exists, but is narrow and triplicated
`ConversationContactState` + `ConversationRuntime` is a **real actor**: out-of-order, multi-day, restart-durable reply routing via a correlation key (`awaitingRunId` → `findByAwaitingRun`, `conversationService.ts:198-202`); its docstring literally says a conversation "rests between messages, resumes when the contact replies days later, across restarts" (`conversationRuntime.ts:1-15`). But it is locked to **one script per App** and **two event kinds** (`onReply`, `onWorkflowComplete`) — the entry/reply unions are closed (`conversationScript.ts:22-68`). Meanwhile the *same* subject's durable fields live in `app_contacts` (`schema.ts:2602-2634`) and its in-flight compute lives in `WorkflowRunState` (`workflow.ts:1290-1308`). → *The lead is ~90% of a virtual object, cut into three tables and usable for exactly one scripted flow.*

### F3 — Four schedulers, none general
`appOrchestrator.sweepSchedules` (`appOrchestrator.ts:200-237`) wakes *workflows* on cron. `ProactiveFollowupService.sweep` (`proactiveFollowups.ts:50-72`, wired at `bootstrap.ts:1421-1464`) wakes *contacts* whose `nextTouchAt` is due — but sends a **fixed template sentence** (`proactiveFollowups.ts:126`), not a specialist's judgment. `CommandHeartbeat` wakes *manager agents*. `ListenerRuntime` wakes *workflows* on events. Each is a correct, restart-safe sweep; none can express "wake **this** agent, or **this** subject, on **this** clock or event, and let it think." → *"Where is the second message triggered?" — the sweeper fires; the intelligence was never plugged into it.*

### F4 — Connections are unowned and sending is unenforced
`agentis.channel.send` lists **all** workspace connections and delivers, never comparing `ctx.agentId` to the connection's owner (`agentisToolHandlers/channel.ts:79-150`; `deliverToConnection`, `channelBridge.ts:217-238`). `channel_connections.agentId` is `notNull` but used only for **inbound** routing (`schema.ts:1070-1099`). `credentials` has no `agentId` column at all (`schema.ts:108-122`). **Any agent can send on any connection.** Yet the exact grant shape the operator wants — `mode: full_delegated | agent_decides | human_approval`, scopes, `expiresAt`, budget — already ships as `groundingAgentGrants` (`schema.ts:2412-2430`), enforced in `contextComposer.ts`, and again as `personalBrainGrants` (`schema.ts:1581-1587`). `InvokeCtx.callerAgentId` is threaded through every capability invocation but used **only for audit** (`capabilityRegistry.ts:188`). There is no deterministic-vs-agent sender identity: a workflow send today can only go through an LLM `agent_task` calling the same unchecked tool (`outboundPolicy` distinguishes only `source: agent|operator`, `outboundPolicy.ts:30`). → *The ACL is built; it guards the wrong door, and the send door has no lock.*

### F5 — Three authoring DSLs; agent-first orchestration is specced, not built
Graph JSON, conversation-script JSON, and listener config are separate authoring systems referencing each other by opaque ID. The engine *can* mutate a live graph — `evolveGraph` with a "green ratchet" (`WorkflowEngine.ts:3034-3199`) and `pursue`/`converge` durable loops exist — but only **within** an already-running graph, against a closed `SUPPORTED_NODE_KINDS` enum (`validateGraph.ts:28-78`) under an authority ladder (`resolveEvolutionAuthority`, `WorkflowEngine.ts:3071`). `AGENT-PRIMARY-POSTURE-ARCHITECTURE.md` already specs "the agent owns the loop, the graph is the plan it evolves" — **specced, not built.** → *"Fixed nodes" is true for authoring and false for execution; the seam to invert it exists and is unused.*

### F6 — Measurement is absent or qualitative
There is **zero** experiment/variant/assignment primitive anywhere in `apps/api` or `apps/web` (grep-verified across both trees). `runAnalytics` computes a real success rate (`runAnalytics.ts:82-229`) but keyed on *run* identity, not any variant or message-position dimension. `appLearning` deposits **text lessons** on outcome transitions (`appLearning.ts:39-391`), not counters — no `GROUP BY outcome` exists. `SWIFT`'s verdict engine is, in effect, a code-based+model-based **grader pair** already — but it is used as a gate, never as a reusable measurement substrate. → *The A/B ask needs one variant field + one aggregation service, not "great engineering from scratch"; the graders already exist.*

### F7 — The platform cannot describe itself, and its names lie
`PLATFORM_KNOWLEDGE`/`PLATFORM_ARCHITECTURE_KNOWLEDGE` (the object ontology) are **explicitly withheld from `mcp_native` harnesses** in `orchestratorPrompt.ts` — the external agents with the *least* context get the *least* guidance. `routes/mcp.ts` flattens every `AgentisError.code/remediation` to `-32603 err.message`, discarding ~60 already-written remediations. The suite path emits the bare enum `` `dry verdict ${dryVerdict}` `` (→ "dry verdict hollow"). And the tree is full of **false affordances** the audit catalogued: `AgentSessionService` "persistent" (per-run only); `appStaffing` "birth-staff" (metadata only, nothing runs); `agents.status` (connection health, not liveness); `channel_connections.agentId` (inbound-only); `callerAgentId` (audit-only); `persistent_listener` (a workflow trigger, not agent persistence). → *An agent cannot query what exists, is told the wrong thing by names, and never receives the remediation the platform wrote for it.*

**Every field failure maps here.** Four duplicate apps = F7 (illegible) + no find-or-create. Un-wireable pipeline = F1+F2+F3 (no persistent agent, no general subject, no general scheduler). "dry verdict hollow" = F6+F7. Mystery approval gate = F7 (doctrine invisible). "No place for persistent agents" = F1. "Any agent shouldn't have connection access" = F4.

## Part 2 — Design principles (2025–2026 state of the art)

The external research converges on a small set of principles that Agentis half-implements. Each below names its sources and the Agentis seam it governs.

1. **Keyed, single-writer durable entity with an inbox.** Restate *Virtual Objects* (per-key single-writer + queued inbox), Cloudflare *Durable Objects* (one id = one thread + storage + alarms), Temporal *Entity Workflows* (one execution per workflow-id, signals durably queued), DBOS (workflow-id = idempotency + address) are the **same shape**. Identity = key; all mutations serialize through one logical thread; pending external input **queues** rather than racing. → governs F1/F2. [restate.dev/vs/temporal] · [developers.cloudflare.com/agents/concepts/agent-class] · [temporal.io/blog/very-long-running-workflows]
2. **Durable suspend/resume by correlation token, not by channel.** Restate awakeables, LangGraph `interrupt()`+thread-id, DBOS `send/recv`, Temporal signal-by-id: mint a unique token, hand it to the outside world, match on it when it returns. **Out-of-order across subjects is a non-problem** because each wait-token is unique. → governs F2. [docs.restate.dev/develop/ts/awakeables] · [docs.langchain.com/oss/python/langgraph/interrupts]
3. **Wake is cheap and multiplexed; the agent is dormant data, not a process.** No credible system holds an OS thread for a multi-day wait. A durable record + a wake condition + a shared dispatcher reconstitutes behavior on demand. **You do not need one process per lead.** → governs F1/F3. [developers.cloudflare.com/agents/concepts/long-running-agents] · [dbos.dev/blog/why-postgres-durable-execution]
4. **Structured, agent-editable memory beats transcript replay; consolidation is a separate async concern.** Letta core-memory *blocks* (labeled, editable, shareable) + *sleep-time* consolidation; Anthropic file-memory + initializer/worker harness. → governs F1. [letta.com/blog/agent-memory] · [docs.letta.com/guides/agents/architectures/sleeptime]
5. **Code, not chained tool calls, is the composition layer.** Anthropic *Code execution with MCP* (150k→2k tokens, 98.7%), Cloudflare *Code Mode* (one `codemode` tool over a generated typed API), Anthropic *programmatic tool calling* (−37% tokens, +accuracy). LLMs are fluent in TypeScript, not in 70-tool JSON chaining. → governs F5/F7. [anthropic.com/engineering/code-execution-with-mcp] · [blog.cloudflare.com/code-mode]
6. **Declared intent + idempotent convergence beats imperative create.** Stripe idempotency keys, Terraform `import`/`plan`/`apply`, Kubernetes reconciliation + server-side-apply field ownership. Re-declaring an existing resource **binds, it does not duplicate.** → governs F7 + the four-duplicate-apps failure. [stripe.com/blog/idempotency] · [kubernetes.io/docs/reference/using-api/server-side-apply]
7. **Authority is explicit, scoped, and audience-bound per agent — never ambient.** MCP OAuth 2.1 (Resource Indicators RFC 8707 bind a token to one resource; non-passthrough kills the confused deputy; `insufficient_scope` step-up), capability-based security, per-agent identity binding. → governs F4. [modelcontextprotocol.io/specification/2025-11-25/basic/authorization]
8. **Multi-agent parallelism is a budgeted trade with explicit task-boundary contracts.** Anthropic multi-agent research (context isolation; ~15× tokens; vague delegation *causes* duplicate work); OpenAI Agents SDK (handoffs + guardrails); Google ADK (supervisor + shared-state/delegation/explicit-invocation). → governs F5. [anthropic.com/engineering/multi-agent-research-system]
9. **Evaluation/experimentation is infrastructure, not a bolt-on.** Anthropic *Demystifying evals* (task/grader/trial/transcript/outcome; **grade the outcome, not the path**; pass@k vs pass^k; *read the transcripts*); OpenAI acquired Statsig ($1.1B) and Anthropic acquired Humanloop to **own** this layer. → governs F6. [anthropic.com/engineering/demystifying-evals-for-ai-agents]
10. **AX is a measured discipline.** Netlify's "Agent Experience" (access/auth · API design · machine-ready docs); tool descriptions have *measured* effect on success (+5.85pt median across 856 tools). Legibility is a loop, never "done." → governs F7. [biilmann.blog/articles/introducing-ax]

## Part 3 — The unified model: one spine, six primitives

Everything below is *one* substrate — a **Durable Entity spine** — projected into six primitives an agent can name and compose. New capability lands **in** this spine or explicitly **generalizes** an existing subsystem into it; it never adds a parallel one (Part 7).

### 3.0 — The spine: the Durable Entity (the one unification)
A single keyed, restart-durable, single-writer-per-key record with a typed **inbox** and a **wake clock**, driven by **one dispatcher**. This is the abstraction Restate/Cloudflare/Temporal/DBOS all share, and it is what `ConversationContactState` + `WorkflowRunState.waitingInputs` + `app_contacts` already *are*, three-quarters-built and split.

- **Tables (extend, don't invent):** promote `ConversationContactState`'s row into a general `durable_entities(id, kind, key, workspaceId, appId?, state_json, next_wake_at, lease_owner, lease_expires_at, awaiting_correlation_json, created_at)` and an `entity_inbox(id, entity_id, event_type, payload_json, received_at, consumed_at)`. `app_contacts` becomes a typed *view/kind* of entity, not a rival table; `WorkflowRunState` stays the **transient per-wake execution** the entity spawns (one run per wake) — the audit's sharpest rule: **do not merge run-state and entity-state; relate them by FK** (`project` R1 mapping).
- **Correlation:** generalize `awaitingRunId` → `awaiting_correlation_json {kind, id}` (a run id, a reply-to token, an upload id, a timer). Out-of-order/multi-day falls out for free (Principle 2). This is a *widening* of `conversationService.#findByAwaitingRun`, not a new mechanism.
- **Wake:** one dispatcher sweep (fold the four schedulers of F3 into it) asks "which entities are due (`next_wake_at ≤ now`) **or** have unconsumed inbox rows?" and resumes exactly those. **Lease mandatory:** SQLite single-writer ≠ per-entity mutual exclusion — a `lease_owner`/`lease_expires_at` claim (the pattern already in `CognitivePromotionQueueWorker`/the channel-turn queue) prevents a cron tick racing a webhook wake (R1's #1 trap).
- **Memory:** the entity's `state_json` is a small set of Letta-style editable blocks (`status`, `last_summary`, `next_action`), not a transcript; raw history stays append-only; consolidation runs *after* each reaction, off the reactive path (Principle 4; reuses Brain formation, `project_brain_memory_formation`).

Everything in 3.1–3.2 is a **kind** of Durable Entity. 3.3–3.6 are the primitives that act on it.

### 3.1 — Agent (persistent / resident) — kills F1
A first-class agent that exists between runs. Compose the pieces that already exist:
- **Identity + memory:** `agentIdentity` + `agentMemory` (already durable, `scopeId=agentId`) — no change.
- **Resident session (the missing 10%):** add `AgentSessionService.getOrCreateResident(agentId)` — a session with `runId: null` (the column *already allows null*, `schema.ts:2072-2073`; chat sessions already use it). Now a woken agent has real persona/task/plan continuity instead of empty history (F1's amnesia).
- **Residency schedule:** generalize `CommandHeartbeat`'s manager-only filter into an opt-in `config.residency {intervalMs|cron, wakeEvents[]}` on any agent — reusing its proven dedup/in-flight/autonomy-double-switch machinery verbatim (`commandHeartbeat.ts:112-142`). An agent becomes a Durable Entity of `kind='agent'` on the shared dispatcher.
- **Owned connections + subject queue:** the agent owns grants (3.3) and supervises N Subjects (3.2) as its work queue.

*This is the same construct as Cloudflare's Agents SDK (`Agent extends DurableObject`) and Letta's addressable stateful agent — one instance, durable state, wakes on schedule/event.* A resident agent and a per-subject entity are the **same primitive at different scope** (R1).

### 3.2 — Subject (durable per-entity actor) — kills F2
Generalize `ConversationContactState`/`ConversationRuntime` from "a contact in one script" to "any subject of any kind, with a typed inbox and independent lifecycle": rename contact→subject; widen the closed `onReply`/`onComplete` unions to a general `onEvent{eventType, branches}` correlated by `awaiting_correlation_json`; keep the datastore persistence (`AppDatastore.upsert/query`, already generic). Each Subject is addressable, waits indefinitely, receives events out of order, and drives its own state machine — the lead, the support ticket, the monitored host, the candidate, the invoice. The narrow `ConversationScript` becomes **one authored behavior** over the general Subject, not the only shape.

### 3.3 — Connection (owned + scoped) — kills F4
- **Grants:** new `connection_agent_grants(workspaceId, connectionId, connectionKind, agentId, scope: read|send|manage, grantedBy, expiresAt)` — a **direct clone of `groundingAgentGrants`** generalized from grounding to any connection (channel, credential, MCP mount). Default-deny.
- **Enforcement:** one lookup at the single send door (`agentisToolHandlers/channel.ts`, where `ctx.agentId` is already in scope) and at `capabilityRegistry.#invokeWithLedger` (where `callerAgentId` already flows) — turn the audit-only identity into an authz gate. Missing scope → structured `insufficient_scope`-style error naming the grant to request (Principle 7).
- **Deterministic sender:** a real pure `channel_send` node (a `pureHandlers` entry calling `ChannelBridge.deliverToConnection` directly, stamped `source: 'workflow_node'`) — token-free, no LLM, distinct identity from an agent. This is the operator's "deterministic sender."
- **Negotiation:** `agentis.connection.request` — an agent that needs a kind of connection it lacks writes an operator-facing request (reuse the approval/`spawn_task` plumbing) instead of silently failing: *"I need a WhatsApp connection bound to me to run this."* (F7's "the agent never asked to connect WhatsApp".)

### 3.4 — Orchestration (agent-first) — kills F5
Build out `AGENT-PRIMARY-POSTURE` on the seam that exists: the agent **owns the loop**; the graph is the **compiled, durable trace it evolves** (`evolveGraph`/`pursue`/`converge` are already the mechanism). Deterministic where a step is proven (a `channel_send`, a data write); agentic where judgment is needed (write message 2, classify the reply). `pursue`/`converge` **is** the orchestrator-worker pattern from Principle 8 — give each spawned worker an explicit task-boundary contract (objective + output shape + tool scope) to kill the duplicate-work failure mode, and a token budget. The three authoring DSLs collapse: a Subject's behavior, a workflow, and a listener become **declarations over the same spine**, authorable as code (3.7 / Part 4).

### 3.5 — Experiment (measurement) — kills F6
A general variant/metric substrate, not an A/B feature:
- **Assign:** an `experiment_assignments(experimentId, subjectId, variant, assignedAt)` row (or a nullable `experimentJson` on the entity, mirroring how `outcome/outcomeAt` were added to `app_contacts`). Any decision point (which message, which model, which connection, which prompt) draws a variant.
- **Measure:** thread `experimentId/variant` through `appLearning.recordOutcome` (it already carries `workspaceId/appId/subjectId/outcome`) so every outcome is variant-tagged.
- **Aggregate:** a ~100-line `experimentAnalytics` sibling of `runAnalytics` — `GROUP BY variant, outcome` → success rate per variant, with **pass@k vs pass^k** and **delta-vs-live** (Principle 9), not just absolute scores.
- **Grade with what exists:** generalize the **SWIFT verdict engine** into the reusable grader (code-based + LLM-judge + human) so *outcomes are graded, not self-reports* (Principle 9). Render per-variant funnels with the grammar that already exists (`PipelineFlow` `stageField=variant`, `Chart` `series`) — no new node kinds (F6, Agent-4 seams).

### 3.6 — Interface (mission-control) — kills the "stupid UIs"
Generalize the per-App "living console" into a cross-agent command center, in the **same declarative AG-UI grammar** (extend the node vocabulary; never arbitrary React — consistent with `AGENTIC-SYSTEMS-ARCHITECTURE §3A`). Three lenses over the spine, all live via the existing `DATA_CHANGED`/realtime bus:
- **Agents** — every resident agent, its residency clock, its owned connections, its subject queue depth, its needs-you flags (the LangChain *Agent Inbox* pattern: accept/edit/respond/ignore over long-horizon interrupts).
- **Subjects** — the pipeline board (found → contacted → replied → won) as a view of `durable_entities` of a kind, filterable, drill-through to the live thread.
- **Experiments** — per-variant success dashboards (`KPIStrip` + `PipelineFlow` + `Chart`), transcript drill-down (never a score without its transcript, Principle 9).

### 3.7 — The build surface: code-mode over the six primitives — kills F7's illegibility
Generate a typed SDK from the existing `agentisToolHandlers/*` module boundaries (Zod schemas already define I/O) and expose **one** `agentis.code.execute({code})` that runs agent-written TS in a sandboxed Node `vm` against the SDK, with the ~70 discrete tools kept underneath for discovery/fallback (deferred loading + `agentis.tool.search`). Composition — loops, conditionals, "find-or-create then wire" — happens in code, where frontier models are strongest (Principle 5). This is the primary answer to "agents can't wire primitives."

## Part 4 — The AX layer (make the six primitives legible)

Structure beats prose (`project_paved_road_agent_experience`). Ship these so the primitives are *discoverable, safe, and self-correcting*:

- **Honest errors (F7).** `routes/mcp.ts` returns `AgentisError.code/remediation/details` as structured MCP content with `isError:true` — instantly surfacing ~60 written remediations. Kill the bare-enum verdict; emit SWIFT's evidence shape (which check, which floor, `got <value>`, producing node, next call).
- **Orientation (F7).** `agentis.orient` + an MCP **resource** `resource://agentis/object-model`: the six primitives + their relationships, the canonical creation order, **the caller's current inventory** (grounded generation → find-or-create), and the disclosed v1 ceilings (`LIMITS.md`). Re-callable; footer-linked from every error. Stop withholding the ontology from `mcp_native`.
- **Find-or-create everywhere (F7, Principle 6).** Idempotency key on every creating tool (`idempotency_keys` table, Stripe-style) + natural-key lookup returning `{created:false, existing:true}` (Terraform-import) + a `plan/diff` before destructive ops (K8s). Generalize the existing blessed-graph/"warn-previously" protection into this universal shape. Restores lifecycle symmetry (ship `app/workflow/agent.archive|delete` wrapping the REST routes that already exist, with `destructiveHint` + terraform-style preview).
- **Annotation quad (Principle 2/7).** Every tool declares `readOnly|destructive|idempotent|openWorld`; the approval layer *enforces* them (HITL escalation is exception-based, not approve-everything). Doctrine-inserted gates stamp `insertedBy` and say so (kills the mystery approval button).
- **Ontological catalog + tool search (F7).** Add an `object: agent|subject|connection|orchestration|experiment|interface|platform` dimension to every tool; deferred-load + search once schemas exceed ~10% of context (85% token cut, accuracy *up*).
- **The eval-driven tool loop (Principle 10) →** Part 5 H3.

## Part 5 — Horizons and sequencing

Ordered by *unblock-the-operator-fastest* then *unify*. H1 is mostly additive/no-schema; each item names the fracture it closes.

### Horizon 1 — Wire the inert, enforce the unenforced, make it legible (days–2 wk)
1. **Plug intelligence into the follow-up sweep (F3).** Replace `ProactiveFollowupService`'s canned template with a dispatch into the App's staffed specialist (judgment from the subject's state) — the operator's "second message" now actually thinks. *(one service, no schema)*
2. **Enforce connection ownership (F4).** `connection_agent_grants` + one check at the send door + `channel_send` deterministic node + `connection.request`. *(clone of `groundingAgentGrants`)*
3. **Resident session (F1).** `getOrCreateResident(agentId)` + generalize `CommandHeartbeat` to opt-in residency. *(nullable-runId seam)*
4. **Honest MCP errors + kill bare-enum verdicts (F7/F6).** *(two files)*
5. **`agentis.orient` + object-model resource + deliver the withheld ontology to MCP agents (F7).**
6. **Find-or-create + idempotency keys + lifecycle symmetry (F7).**

### Horizon 2 — Collapse to the spine (2–8 wk)
7. **The Durable Entity spine (F1/F2/F3).** `durable_entities` + `entity_inbox` + one dispatcher + mandatory lease; fold the four schedulers in; generalize `awaitingRunId`→correlation key.
8. **Generalize Subject (F2).** Widen the closed script unions to `onEvent`; `app_contacts` and `ConversationContactState` become kinds/views of the spine.
9. **Agent-first orchestration (F5).** Build out `AGENT-PRIMARY-POSTURE` on `evolveGraph`/`pursue`; task-boundary contracts + budgets on worker fan-out.
10. **Experiment substrate (F6).** Assignment + variant-tagged outcomes + `experimentAnalytics` + SWIFT-as-grader.

## Part 6 — Success criteria

**The operator's stress test must fall out of the six general primitives with zero use-case code:** a resident **Agent** (3.1) on a 5-min residency wakes, runs discovery, mints **Subjects** (3.2); a deterministic `channel_send` (3.3) sends the token-free greeting on a connection **owned** by that agent; each Subject waits out-of-order for days on its inbox; on reply the agent (agentic **Orchestration**, 3.4) writes and classifies; a positive reply triggers build-and-deliver, then the Subject reaches terminal and stops; message variants are an **Experiment** (3.5) with live per-variant success rates on the **mission-control** interface (3.6) — the whole thing authored in code-mode (3.7).

**Generality proof (the real bar):** the *same six primitives*, unchanged, express — a **support desk** (Subjects = tickets, Experiment = macro variants), an **infra-monitor** (Subjects = hosts, inbox = alert events), a **research agent** (resident Agent + web connection + artifact outcomes), a **company automation** (many resident Agents + shared Subjects + scoped connections + HITL approvals), and a **coding-agent fleet** (Subjects = issues/PRs, graders = SWIFT/tests). If any of these needs a new subsystem, the abstraction failed.

**Measured by the AX harness (Part 5 #12):**

| Milestone | Criterion |
|---|---|
| H1 done | A cold external agent states the six-primitive model, cleans up a seeded duplicate app, and every error it hits carries a remediation. The follow-up sweep sends a *thought* message; sending on an unowned connection is refused with a scope error. |
| H2 done | "resident agent → discover subjects → deterministic send → wait-reply (out of order, days) → classify → build → deliver → stop" runs on the spine with **zero duplicate resources** and **one** durable-state model; the same run survives an API restart mid-wait. |
| H3 done | AX eval suite green in CI (duplicate-count 0 across tasks); ≥80% reduction in median tokens-to-working-system vs today; one canonical `ARCHITECTURE.md`; per-variant success dashboards render on real outcome data. |

## Part 7 — What NOT to do

- **No use-case patches.** No "outreach template," no "lead pipeline node." The lead system must fall out of general primitives (3.1–3.6), or the fix is wrong.
- **No ninth subsystem — this is the cardinal rule.** New capability lands *in* the Durable Entity spine or *explicitly generalizes* a named existing subsystem into it. Adding a parallel spine is the disease this plan cures (`feedback_no_duplication`).
- **Don't rebuild Temporal/Restate.** Adopt the *pattern* (keyed entity + correlation-token suspend + queued inbox + lease) on SQLite/Node; the infra of those systems is overkill for single-tenant OSS.
- **Don't conflate run-state with entity-state.** A run is the transient thing a Subject spawns per wake; keep them separate tables with an FK (R1's sharpest trap).
- **Default-deny authority; never ambient.** No auto-broad grants at mount; every grant is an audit-visible action; check per hop (confused-deputy).
- **Grade outcomes, never self-reports.** Every gate carries executable evidence; read transcripts before trusting a score.
- **The DSLs, canvas, and interface are *views* over the durable core** — never the durability boundary (OpenAI's Agent Builder deprecation).
- **No V2 suffixes; "Goal" stays reserved** for the future long-term tier — run/subject-scoped acceptance is "Objective" (`feedback_architecture_naming_framing`). Append to the §9 log as code lands (`feedback_masterplan_log`).
