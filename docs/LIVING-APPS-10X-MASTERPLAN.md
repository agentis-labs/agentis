# Living Apps — the Agentic App becomes a place an agent *lives*, not just a tool it *uses*

> **Status:** Proposed · 2026-06-26
> **Extends (does not fork):** `AGENTIC-APPS-10X-MASTERPLAN.md` (the App primitive), `OMNICHANNEL-ORCHESTRATOR-10X.md` (the channel→turn loop), `GENUI-RENAISSANCE-MASTERPLAN.md` (the expressive interface layer). This plan does not rebuild any of them — it **joins** them.
> **One line:** *An Agentic App is a living digital worker you instantiate: it perceives across many channels, thinks with a memory and skills that compound, acts, expresses itself as a calm operator console, and gets measurably better over time — staffed by a team that is born with it.*

---

## 0. The realization

Agentis already builds **automations** well: something happens, the App's workflow runs, the datastore updates, a surface shows it. But the agent era's other half is the agent that is simply **present** — an online sales attendant who greets every visitor, qualifies them, remembers the objection from three days ago, follows up Thursday as promised, and escalates to a human when the deal is real.

Agentis already *has* that resident agent. **It just doesn't live in the App.** It lives on a parallel spine the App cannot see. The question that opened this plan — *"is the App the right home for an online agent, or does the agent merely use the App as a tool?"* — has an honest answer in the code: today it's the latter, by accident. This masterplan makes the App the home.

---

## 1. The gap, grounded

Agentis has **two disjoint spines** that never meet.

**Spine A — the resident conversational agent** (real, working, App-blind):
```
channel_connections (agentId — NOT appId)         schema.ts:1015
   → ChannelBridge.handleInbound
   → ChannelTurnDispatcher.dispatch()              channelTurnDispatcher.ts
        → resolve the connection's bound agent
        → ChatSessionExecutor.turn()               (the agent actually thinks)
        → conversations (agentId, channelConnectionId)  schema.ts:810
        → deliver the reply to the channel
```
A genuine persistent agent — long-lived threads, a 20-message window (`HISTORY_LIMIT`), Brain memory, cross-channel identity recall, sticky permission mode. **It has no concept of an App:** `channelTurnDispatcher.ts` has zero `appId` references; a channel binds to an *agent*, a conversation is owned by an *agent*. Neither can point at an App.

**Spine B — the Agentic App** (the primitive, conversation-blind): `apps → workflows + collections + surfaces + app_members + ownerAgentId` ([schema.ts:2653](../packages/db/src/sqlite/schema.ts)). `routes/apps.ts` has zero `conversation` references. The App cannot see, own, or render the live relationship.

**The tell: the UI already dreams of the bridge.** The GenUI grammar already ships the conversational vocabulary — `Inbox`, `ChatThread`, `ConversationThread`, `AgentConsole`, `ActivityStream` are real, renderable nodes. But every one binds to **App datastore rows** via `useBoundRows` ([ViewRenderer.tsx:1552](../apps/web/src/components/apps/ViewRenderer.tsx)), and `send` is a datastore/tool action. An App's "Inbox" today is a **photograph** of conversations copied into a collection — not the live thread, and sending from it runs no real turn.

> The interface vocabulary is ahead of the runtime. The grammar imagines a living conversational app; the substrate behind it is a still life. This plan closes exactly that distance — and then goes past it.

## 2. The model: an App is a composition, not a mode

An App is **not** typed `triggered | resident`. That binary is a false ceiling — it would throw away the plural activation Agentis already has and scatter `if (resident) …` through the codebase. An App is a **living service composed from orthogonal dials**, each independent and mixable:

- **Senses (activation)** — how it wakes: `manual | cron | webhook | persistent_listener` ([workflow.ts:22](../packages/core/src/schemas/workflow.ts)), a channel message, a data change, another agent, or its own standing goal. Already six-plus forms, built (`ListenerRuntime`, `scheduler`, `broadcastDispatcher`, `channelTurnDispatcher`, `TriggerRuntime`, `ActiveWorkflowRegistry`).
- **Persistence** — one-shot → session → long relationship → perpetual-standing.
- **Counterparts** — one human · many humans · systems/data · other agents · the physical world · itself.
- **Surfaces** — chat · operator console · voice · email · public web · API · embed.
- **Staff** — a single agent · a cast · a swarm · a hierarchy.

"Automation" and "resident attendant" are just two points in this space, and most real apps span several at once. New application shapes are **new compositions of primitives that already exist**, never new modes or new code:

| Shape | Senses | Persistence | Counterpart | Surface |
|---|---|---|---|---|
| Automation | event/webhook | one-shot | systems | log |
| Resident attendant (sales/support) | channel + schedule | standing | one human | console + chat |
| Monitor / watchdog | data stream + listener | standing | systems | console + alerts |
| Researcher / creator | own goal + schedule | standing | itself + web | artifacts |
| Broadcaster / community | inbound + schedule | standing | many humans | broadcast + inbox |
| Co-pilot | human acts on a surface | session | one human | embedded surface |
| Voice line / field agent | voice/webhook | session→standing | human / world | voice |
| Agent-to-agent service | another agent's call | one-shot→standing | other agents | API |

**The invariant:** there is **no `app.mode` flag anywhere.** The resident console is simply an App whose senses include a channel and whose persistence is standing. Generality and minimalism agree here — the composable model has no branch, so it is both more open and more minimal. *If we ever feel tempted to write `app.mode`, that is the signal we took the wrong turn.*

**What we build first.** The **resident-conversational composition** — because it is the hardest preset (it forces the channel↔App spine, multi-party threads, the live surface) and the one needed now. We build it as the *first instance of the general model*, not as one of two types.

**Constraints we hold throughout:** additive and nullable (an App with no channels behaves exactly as today); one render path (edit = live = operator = public); we extend `ChannelTurnDispatcher` / `conversations` / `view.ts` / `ViewRenderer` / `apps`, never fork a parallel "conversation app" subsystem (`feedback_no_duplication`); and the agent does the wiring — no operator settings panel where a default will do (`feedback_no_friction_defaults`).

## 3. The organism — perceive · think · act · express · grow

The deepest reframe is what an App *is*. Today Agentis assembles a service from **static parts**: a UI artifact + a prompt + a connection. The bolder model treats it as **one organism** — the App is the **body**, Brain + Abilities the **mind**, the surface the **face**, channels the **senses**, and graduation the capacity to **grow**. That is the difference between "AI-powered CRM apps" and *the substrate where you instantiate a digital worker that gets measurably better* — the actual #1-platform claim. Four design principles follow.

### 3.1 The interface is *performed*, not authored
The surface is a **live projection of the agent's attention**, not a frozen `app_surfaces` row. The `ViewNode` grammar stays (it's good; `CodeSurface` is the long-tail hatch) but becomes a **streaming medium**: a stable, calm frame (inbox · active thread) hosts **agent-composed regions** — when 40 threads share a churn signal, the agent renders a "churn risk" panel *unprompted* and it appears. The UI becomes an output channel of cognition.
*DNA guardrail (alive, not flickering):* stable skeleton + agent-zones only; every agent-composed region is **explainable** ("added because 12 deals stalled at 'pricing'") and the operator can **pin / freeze** it.

### 3.2 The specialist is *grown*, not configured — and abilities are part of the agent
An agent's competence is **persona + its pinned abilities (always composed into context) + context-selected abilities + brain-recalled lessons** — assembled every turn, never a frozen prompt.
- **Pinned abilities are identity.** Today abilities fire **only on an explicit `/slash`** ([chatSessionExecutor.ts:267](../apps/api/src/services/chatSessionExecutor.ts)) even though they can already be *pinned to an agent*. A pinned ability must be composed into the operating context every turn, like the persona; slash becomes a manual override, not the only trigger.
- **Auto-selected abilities** (the Ability Composer, the real remaining gap — deferred in `project_abilities_10x`): mid-conversation the agent reaches for its "price-objection" ability because the customer raised price, composed by relevance.
- **A learning loop:** every won/lost/abandoned conversation deposits a **graded lesson**; recurring winning patterns **graduate into new abilities** (reuse the self-eval + graduation-by-reuse machinery — `project_pacer_feynman`). The operator can *watch the skill grow*.
- **Where competence lives:** facts & relationships → Brain; reusable how-to-operate → Abilities (a first-class facet of the agent); non-negotiable doctrine → the manual. *Memory is already real:* recall is semantic by default (bundled multilingual ONNX `Xenova/multilingual-e5-small`, **no lexical fallback**, model-id-checked — [embeddingProvider.ts:220](../apps/api/src/services/embeddingProvider.ts), [sharedIntelligence.ts:348](../apps/api/src/services/sharedIntelligence.ts)). The residual is not "can it remember" but "does the resident loop **feed and scope** that memory" (per-conversation/contact lessons + contact-scoped recall — G11).

### 3.3 A resident App is staffed by a *cast*, not one agent
A desk is a team — greeter, qualifier, objection-handler, closer, scheduler — each a composable specialist you can hire, clone, train, and swap, **handing off inside one thread** (which is *why* multi-party threads, G1, are load-bearing). The org spine exists (`project_subdomains_specialist_responsibility`); it isn't yet composed into a living team that improves part-by-part.

### 3.4 Specialists are normal agents; what's missing is *creation*
A specialist is a **normal Agentis agent**, reusable across apps and any task — no app-imprisonment, no residence binding. The workspace talent pool and `SpecialistDemandRouter` stay. `app_members` is already a many-to-many join, so one agent already *can* serve many apps. The failure is at birth: `AppStore.create` ([appStore.ts:95](../packages/app/src/appStore.ts)) sets `ownerAgentId: null` and inserts **zero** `app_members`; the `app.create` tool passes no staff. **An App is born unowned and empty of people**, even though `SpecialistAgentService`, `specialist_loadout` (role→abilities), and the membership join all already exist. The fix is one new step — *birth-staff-at-creation* (§4.6) — not a new ontology. Everything else org-side (managers own subdomains; an app's own staff answers for app work first) is **doctrine + a precedence tweak**, never a schema constraint.

## 4. Architecture

### 4.1 Ownership — the App adopts channels and conversations *(additive, nullable)*
- `channel_connections.app_id → apps.id (set null)` — a channel can belong to an App.
- `conversations.app_id → apps.id (set null)` — a thread can belong to an App; backfilled from the connection on inbound.
- New table `app_contacts` — the **relationship** entity the App is missing: one row per person, unifying their conversations across channels (keyed off `channel_identities`). Fields: `appId, displayName, primaryChannel, handles[], stage, goal, lastTouchAt, nextTouchAt, dataJson`. This is what turns a pile of threads into a pipeline.

`channel_connections.app_id` makes the App the unit you **connect, deploy, pause, and hand off** — "connect this WhatsApp line to the Sales App" is the mental model.

### 4.2 The resident turn — context injection, not a new engine
When `connection.appId` is set, `ChannelTurnDispatcher` runs the *same* turn it already runs, with an **App-scoped context** added: the agent's tool catalog gains the App's datastore (`agentis.appData.*`), its actions, and its operating doctrine ("you are the resident agent of *Acme Sales*; persist what you learn to `contacts`/`deals`; your goal for this contact is…"). The turn writes to the datastore as durable working memory (`project_agentic_apps_data_loop`), and the reply streams to **both** the channel and the App's console via the realtime bus the dispatcher already publishes on. *No new turn engine — "run the turn I already run, but in App context, and broadcast it to the console."*

### 4.3 Live conversational nodes *(additive grammar)*
`Inbox`, `ChatThread`, `ConversationThread` gain an optional `source`:
- `source:'collection'` — today's behavior, unchanged (datastore rows).
- `source:'conversations'` — binds to the **real** `conversations`/`messages` spine, scoped to the App; filters by channel/contact/stage/unread; `send` runs a **real agent turn** (or posts an operator message) through the dispatcher.
The renderer gets `useLiveConversations(appId, filter)` (parallel to `useBoundRows`). The agent authoring the surface picks the source; the operator never configures it.

### 4.4 The datastore as shared working memory
Resident agent and surfaces read/write the same typed collections: the agent learns "budget $40k, decision Q3" → writes `contacts`/`deals` → the pipeline board updates live → a follow-up workflow reads `nextTouchAt`. One memory, three readers — the data loop now spanning conversation *and* automation.

### 4.5 Proactivity — follow-ups as first-class
A resident agent must **initiate**. `app_contacts.nextTouchAt` + the existing `SchedulerService.sweepDue` (`project_live_workspace_steps_10x`) → at the due time, dispatch a turn into the contact's conversation ("follow up as promised"); the agent composes and sends. The machinery exists — it just needs binding to a contact instead of a workflow node.

### 4.6 Birth-staff-at-creation
The App creation pipeline gains a **`staff` step**: for each role the app needs, **reuse a fitting workspace agent** if one exists, else **materialize a new specialist** via `SpecialistAgentService`; **pin its abilities** from `specialist_loadout`; **seat it in `app_members`** (one `operator`, the rest `worker`); set `apps.ownerAgentId`. The cast is **derived from the app archetype** (Sales → greeter/qualifier/closer; analytics → one operator-analyst) and editable after — zero-config. A born specialist arrives with **competence** (composed abilities, §3.2), not merely an extension list. A small precedence tweak lets `resolveResponsibleSpecialist` consult `app_members` first for app work; the org canvas relabels managers as Domain/Subdomain owners and apps as where specialists work — no migration, `domains.managerId` stays permissive.

## 5. The creation doctrine — the building agent must *know* it can do all this

None of the above matters if the agent that builds apps doesn't know these shapes exist. App-building knowledge lives in `orchestratorPrompt.ts` (`PLATFORM_KNOWLEDGE`, the numbered build steps in `PLATFORM_ARCHITECTURE_KNOWLEDGE`, `ORCHESTRATOR_BEHAVIOR_RULES`) and the imported `WORKFLOW_DESIGN_DOCTRINE` — the same place that already teaches "an App = {identity, surfaces, logic, data, agents, policy}" and "2b. CLOSE THE DATA LOOP." Today that doctrine describes only the **automation** shape and says nothing about senses, resident apps, staffing, or the possibility space. We extend it (and mirror it into `agentOperatingManual.ts` so resident agents carry the same self-knowledge, and into the creation pipeline like `WORKFLOW_DESIGN_DOCTRINE`):

- **An App can be any composition (§2), not just a triggered workflow.** Teach the dials: choose the senses (channel / schedule / listener / event / self-goal), the persistence, the counterpart, the surfaces. Give worked recipes — "a 24/7 attendant," "a monitor," "a broadcaster" — so the agent reaches past the automation default.
- **Staff the App at creation (§4.6).** When building an App that operates continuously or talks to people, **assemble its team**: reuse or create specialists, pin their abilities, seat them, set the owner — *with abilities, not just extensions*. An app that operates a relationship but has no resident staff is incomplete, exactly as "logic with no interface/data" is already called incomplete.
- **Close the *relationship* loop, not only the data loop.** The existing "persist results to the collection the UI binds to" doctrine generalizes: a resident agent persists what it learns to `app_contacts`/`conversations`, sets `nextTouchAt`, and renders the live `source:'conversations'` console — so the interface reflects the living relationship, not a snapshot.
- **Compose abilities into the agents you create.** When authoring a specialist for an App, draw its loadout from `specialist_loadout` / the ability library and pin it — the agent is born competent.
- **Still one App, edited in place.** All current anti-duplication doctrine holds (find-and-edit the existing App; `app.create` is idempotent) — staffing and senses are added to the *same* App, never a renamed clone.

**Net:** the agent should treat "what kind of living thing should this App be, and who should staff it?" as part of *every* build — the way it already treats "what's the data model and does the loop close?"

## 6. The experience — brilliant, and ruthlessly minimal

> One object, one place; the agent does the wiring; the operator watches, directs, approves. A living thing you can glance at and trust — not a dashboard with forty knobs.

**One object, one place.** A resident App opens to a **single living console**, not a settings tree. The operator opens *Acme Sales* and it is already alive: threads left, the active relationship center, the agent's presence and pipeline right. The App is the agent's home; the operator is a guest who can step in.

**The console — composed, never crammed** (auto-authored, like every surface):
```
┌─ Toolbar:  Acme Sales · ● 12 live · 3 need you ──────────────────────────┐
│  Inbox (live, source:conversations)   │  Active thread        │  Rail     │
│  ─ Maria · WhatsApp · "still thinking" │  ┌─ contact card ──┐  │ ┌Pipeline┐│
│  ─ João · Instagram · qualifying  ●    │  │ João · $40k · Q3│  │ │ ▓▓▓░ 8 ││
│  ─ Ana · Site chat · ready to buy  ◆   │  └─────────────────┘  │ │ Funnel ││
│                                        │  ‹live messages›      │ ├────────┤│
│                                        │  ‹agent is typing…›   │ │Activity││
│                                        │  [ Take over ] [Note] │ │ stream ││
└────────────────────────────────────────────────────────────────────────────┘
```
Every element is an existing node (`Inbox`, `ChatThread`, `AgentConsole`, `Funnel`, `DataBoard`), now on live data. The taste engine (`genui.ts`) gains a **`relationship`** archetype that composes this automatically when an App has channels + a contact-shaped collection — no template to pick.

**Conversational construction.** Building an interface should feel like talking to a designer who renders as you speak: "show only deals over $20k," "put the funnel above the activity feed." Each utterance is a `SurfacePatch` (the patch grammar already exists — [view.ts:377](../packages/core/src/types/view.ts)) applied live. Conversation is the primary authoring surface; the visual editor is the fallback.

**The human moment — handoff, takeover, approval** (the single most important UX):
- **Take over** — one tap; the agent goes quiet, the operator types into the *same* thread, the customer sees no seam; tap again to hand back with a one-line brief the agent reads.
- **Needs-you** — the agent *flags*, doesn't interrupt ("Ana's ready to buy, wants a discount I can't approve"); it surfaces as `◆` and feeds the `/home` Live Workspace "needs attention" lane.
- **Approve** — outbound that crosses a policy line becomes a one-line in-thread confirm, reusing `interpretConfirmation` + the approval inbox.
This needs multi-party threads (G1) — the linchpin.

**Watch · Build · Run** — three operator lenses on one shell (`AppRuntime.tsx`), never more: *Watch* (glanceable live console), *Run* (act inside surfaces — take over, send, approve), *Build* (conversational construction + visual editor). The customer-facing side is the *same* surface tree published via the existing share-token path, scoped to one contact — one render path, two audiences.

**DNA guardrails:** zero-config or it doesn't ship · one render path · the agent does the wiring (never ask the operator for a `matchField`) · glanceable before deep · additive forever (every existing App renders byte-identically).

## 7. Deeper gaps — what the current design *cannot* express

Real architectural ceilings, ranked by how badly an agentic solution needs them.

- **G1 · Multi-party threads.** `conversations.agentId` is **notNull and singular** ([schema.ts:819](../packages/db/src/sqlite/schema.ts)). A real desk needs customer + resident agent + escalation specialist + human in **one** thread. **Fix:** a `conversation_participants` join, `agentId` demoted to "primary." *The single highest-leverage change.*
- **G2 · Durable turns at scale.** The dispatcher is fire-and-forget, in-process (`void this.#executeTurn`) — no worker pool, no backpressure, no resumption on restart (`project_specialists_seam` deferred turn-durability). **Fix:** a durable turn queue (`DURABLE-TASK-SPINE-10X`) with at-least-once delivery + per-App concurrency.
- **G3 · The relationship has no home.** Threads are unified by channel handle, never by person. **Fix:** `app_contacts` (§4.1) with cross-channel identity merge.
- **G4 · Long-horizon memory.** `HISTORY_LIMIT=20` + recall, but no rolling per-conversation summary — the agent forgets the middle of a month-long deal. **Fix:** per-conversation summarization (Brain formation machinery exists — `project_brain_memory_formation` — just unbound to live threads).
- **G5 · Agent-initiated outreach.** Everything routes from an inbound webhook; nothing lets an agent open a thread proactively. **Fix:** an outbound dispatch path symmetric to the inbound one (the bridge already *delivers*; what's missing is the agent *initiating*).
- **G6 · Voice / real-time media.** Text webhooks only; no streaming-audio channel kind or barge-in turn model. **Fix:** a new channel tier — large; own masterplan, named now so the channel abstraction doesn't exclude it.
- **G7 · Outbound safety envelope.** No per-App rate limit, outbound budget, claim-allowlist, or "never promise X" beyond the generic confirm gate. A commercial agent can't ship without it. **Fix:** per-App outbound policy in `apps.policyJson` (the column exists; enforcement doesn't).
- **G8 · No rehearsal.** You can test workflows, not a multi-turn conversation against the resident agent. **Fix:** a conversation simulator (a synthetic counterpart driving scored scenarios).
- **G9 · Co-presence.** No real-time "we're both on this thread now" (operator sees customer typing, etc.). The bus can carry it; the model doesn't express it. Tied to G1.
- **G10 · The specialist can't learn to operate, and its own abilities are dormant.** Static operating manual ([agentOperatingManual.ts:16](../apps/api/src/services/agentOperatingManual.ts)); abilities slash-gated ([chatSessionExecutor.ts:267](../apps/api/src/services/chatSessionExecutor.ts)). **Fix:** pinned abilities as always-on identity + Ability Composer + the graduation loop (§3.2).
- **G11 · The mind remembers, but the resident loop doesn't feed or scope it.** Recall is semantic and real ([embeddingProvider.ts:220](../apps/api/src/services/embeddingProvider.ts)) — the residual is per-conversation/contact lessons aren't written from threads and recall isn't contact-scoped. **Fix:** bind brain-formation to the live thread + scoped recall. A wiring job over working machinery.
- **G12 · The UI is a stored artifact, not a performed projection.** **Fix:** an agent-driven `SURFACE_RENDER` region channel over the existing realtime seam (§3.1).
- **G13 · Apps are born empty of people.** `AppStore.create` staffs nothing despite `app_members` + `SpecialistAgentService` + `specialist_loadout` existing. **Fix:** birth-staff-at-creation (§4.6) — the narrowest, highest-leverage gap in the org story.

> G1, G2, G3 are **load-bearing** — without them "resident App" is a demo. G4–G13 are the difference between a product and *the #1 platform of the agent era*.

## 8. Phasing

> **Spine track** (own + render the living relationship) and **Mind track** (make the specialist grown) run in parallel. The Mind track is unblocked — semantic brain recall already ships.

- **Phase 0 — App owns the relationship.** `app_id` on `channel_connections` + `conversations`; App-scoped turn context; data loop spans conversation. *→ a WhatsApp line on an App answers in App context and persists.*
- **Phase R — Birth-staff-at-creation (G13).** The `staff` step (reuse-or-materialize agents, pin `specialist_loadout` abilities, seat in `app_members`, set owner) + precedence tweak + org-canvas relabel + the creation doctrine (§5). *Small, self-contained, high-leverage — proves §3.2/§3.3/§3.4 at once.* *→ every new App is born owned and staffed, agents arriving competent.*
- **Phase 1 — Live conversational surfaces.** `source:'conversations'`; `useLiveConversations`; the `relationship` archetype. *→ open the App, watch live threads stream in.*
- **Phase 2 — The human moment (needs G1).** `conversation_participants`; take-over/hand-back; needs-you flags; in-thread approval. *→ seamless human step-in.*
- **Phase 3 — Relationship + proactivity.** `app_contacts`; identity merge; `nextTouchAt` → proactive follow-up; pipeline board. *→ a desk that remembers and follows up.*
- **Phase M1 — Abilities become part of the agent (G10).** *Step 1:* pinned abilities composed every turn (un-gate `/slash`). *Step 2:* the Ability Composer (auto-select by relevance). *→ the agent applies its own skills unbidden.*
- **Phase M2 — The learning loop (G10).** Won/lost → graded lesson → graduated ability; a "learned this week" panel. *→ measurably better over time, visibly.*
- **Phase 4 — Conversational construction, finished.** Live `SurfacePatch` dialogue as the primary authoring path.
- **Phase M3 — Performed surfaces + in-thread handoff (G12, G1).** Agent-driven `SURFACE_RENDER` regions (stable frame + pinnable); the cast hands off in one thread. *→ the console composes itself; the team hands off seamlessly.*
- **Phase 5 — Durability + safety + rehearsal (G2, G7, G8).** Durable turn queue + per-App concurrency; outbound policy; conversation simulator. *→ ships to real money, 24/7.*

**Deferred / own masterplans:** voice channels (G6), long-horizon summarization binding (G4), full co-presence (G9).

## 9. Non-goals
- No new chat *engine* — the resident composition is context injection over `ChatSessionExecutor`/`ChannelTurnDispatcher`.
- No `app.mode` flag, ever — App shape is a composition of dials (§2).
- No parallel "conversation app" entity — Apps absorb the spine; we don't fork it.
- No agent imprisonment — specialists stay normal, reusable agents; no `home_app_id`.
- No operator-facing wiring config — the agent authors; defaults derive.
- Multi-tenant authz stays out of scope (OSS single-operator — `project_masterplan_2026_06_audit`).

## 10. North star
> An Agentic App is not a tool an agent uses — it is a **living digital worker you instantiate, born with its team, and it grows.** Give it senses and it wakes: it perceives across every channel, thinks with a memory and skills that compound, acts through workflows and tools, expresses itself as a calm console that composes around what it's seeing, and gets measurably better every week — as a cast, not a script. You watch, direct, and approve; it does the work, the wiring, and the learning. Fix the App's *ownership and lifecycle* — let the agent live there, be staffed there, and learn there — and the App becomes the home an online agent deserves, and Agentis becomes the substrate the agent era is built on.
