# PAVED ROAD — AGENT EXPERIENCE 10X

> **Status:** PLAN → implementing · **Date:** 2026-07-02 · **Owner:** platform
> **Thesis:** After BUILD-LOOP + UNBREAKABLE + AGENT-PRIMARY shipped, Agentis has world-class *machinery* and a hostile *experience*. Agents are lost not because tools are missing but because **(a) two exposed doors bypass every contract organ, (b) every tool result is terminal — no gradient, (c) the loop has no persisted state to orient against, (d) doctrine never reaches external MCP clients, (e) in-run agents can't see the loop, and (f) when a run dies nobody is woken.** Fix = make the paved road the *only* road, and make every step of it hand the agent the next step.
>
> Evidence: direct code audit 2026-07-02 (4 parallel subagent audits died on session limits; findings below are first-party, each cited to file:line and re-verified).

---

## 0. Verdict

The last three masterplans built the contracts (couplings, intent manifest, green ratchet, failure taxonomy), the test bench (dry_run + assertions + pinned fixtures), the debug loop (debugRun, diagnose, replay) and live evolution (evolve_plan). All shipped, all tested. **And the operator still watches agents get lost.** Why:

An LLM agent is a gradient-follower. At every step it picks the most plausible next action from (tool names + descriptions + last result). Agentis gives it **68 flat tools** ([mcp.ts:157](../apps/api/src/routes/mcp.ts) `tools/list` over `catalog({mcpOnly:true})` — 68 `mcpExposed` entries), **zero server instructions** (the `initialize` result at [mcp.ts:144-148](../apps/api/src/routes/mcp.ts) has no `instructions` field — the MCP-spec mechanism for teaching clients), and **results that end conversations instead of continuing them** (`build_workflow` returns `{workflowId, graph, readiness…}` and never says "now dry_run it"; `workflow.run` returns `{runId, status:'started'}` and never says "poll run.status; diagnose on FAILED"). The doctrine exists — `AGENTIS_BUILD_CONTRACT` — but reaches only platform-spawned chat sessions ([orchestratorPrompt.ts:400](../apps/api/src/services/orchestratorPrompt.ts)), not external harnesses connecting to `/v1/mcp/rpc`.

Worse: the contracts themselves are **optional, keyed on which tool the agent happens to pick**. `agentis.workflow.create` ([build.ts:183](../apps/api/src/services/agentisToolHandlers/build.ts)) inserts a raw graph — no strict validation, no coupling analysis, no intent manifest, no repairs — while sitting beside `agentis.build_workflow` with an equally plausible name. `agentis.workflow.patch`'s at-rest branch ([build.ts:279-293](../apps/api/src/services/agentisToolHandlers/build.ts)) does layout + save, **skipping every organ**. An agent that picks the "wrong" (natural-sounding) door builds exactly the corpse the UNBREAKABLE architecture was written to prevent.

This is not a capability gap. It is an **experience defect**: the platform never tells the agent where it is, what is red, or what to call next — and lets it wander off the road it paved.

## 2. Doctrine — the Paved Road

One road, five steps, and the platform *carries* the agent along it:

```
AUTHOR ──▶ (auto-gates) ──▶ DRY-RUN ──▶ DEBUG-RUN ──▶ RUN (production) ──▶ observe / evolve
   build_workflow          dry_run       run{debugRun}     run                run.status / diagnose / evolve_plan
```

1. **One door.** Every path that persists a graph passes the same gate pipeline (structural → couplings → intent → ratchet). No exposed tool may skip it.
2. **Every result is a signpost.** Every loop tool returns `compass: { stage, workflowId, next: [{tool, args, why}] }` — deterministic, real IDs baked in. The agent never has to remember the doctrine; the road carries it.
3. **The loop is state, not folklore.** `settings.buildLoop` records validated/dry-run/debug-run/production evidence keyed to a graph hash. One call — `workflow.loop_status` — answers "where am I?" for any agent or human.
4. **Doctrine reaches every surface.** MCP `initialize.instructions` delivers the Build Contract + tool-family map to external harnesses; in-run sessions get loop tools; descriptions carry loop-position markers.
5. **Failure wakes someone.** A terminal-FAILED production run files a deduped Issue carrying the diagnosis + exact next calls. Nothing dies silently.

## 3. The plan

### P0 — One door (close the bypasses) *(highest leverage, smallest diff)*
- `agentis.workflow.create` → delegate to `createWorkflowFromDescription` (graphDraft path) so it inherits every gate, repair, App-anchoring, and manifest. Keep the tool id + result shape (compat), add `compass`.
- `agentis.workflow.patch` (at-rest branch) → run the same critical gates before save: `validateWorkflowGraph` + `analyzeEdgeCouplings` + `checkIntentIntegrity` (approval-bypass hard-block; capability-removal block unless `confirmIntentChange`) + Organ-3 regression diff vs prior graph. Reject with named regressions; never persist a regression.

### P1 — The Compass (`services/workflowCompass.ts`) *(the keystone)*
- `graphContentHash(graph)`; `readBuildLoop(settings)`; `stampBuildLoop(db, workflowId, patch)` — persists `settings.buildLoop = { graphHash, validatedAt, dryRun?: {at, ok, issues, hash}, debugRun?: {at, runId, status, hash}, productionRun?: {at, runId, status} }`. A build/patch that changes the hash invalidates stale stamps.
- `compassFor(...)` → `{ stage: 'draft'|'authored'|'dry_run_green'|'debug_passed'|'production', next: [{tool, args, why}] }` — pure, deterministic, no LLM.
- Wire stamps + compass into: `build_workflow` (validatedAt; next=dry_run), `dry_run` (dryRun stamp; next=fix issues | debug-run), `workflow.run`/`ephemeral.run` (next=poll status; diagnose on fail), `run.status`/`workflow.status`/`run.diagnose` (terminal-aware next: diagnose → patch → dry_run → re-run), `workflow.patch`/`evolveGraph` results.
- Engine: at terminal transition, stamp `debugRun` (when `debugRuns` has the run) or `productionRun` on the workflow row — the run itself closes the loop.
- New tool `agentis.workflow.loop_status` (mcpExposed): workflowId → buildLoop + compass + readiness. THE orientation primitive.
- `run.diagnose` gains `nextCalls: [{tool, args}]` — machine-actionable versions of its prose fixes.

### P2 — Doctrine delivery
- `initialize` result gains `instructions` (MCP spec): compact Paved Road card = the 5-step loop with exact tool names + the expression contract essentials + data-flow footguns + one-line tool-family map. Also served on `/server-card`.
- Loop-position markers in the 9 loop tools' descriptions (`[PAVED ROAD 1/5 — author]` …), so the sequence is visible in a flat `tools/list`.

### P3 — In-run parity (unbox the loop)
- `CONTROL_TOOLS` += `dry_run_workflow` + `check_run` — late-bound deps in bootstrap (mirror `evolvePlan` wiring) calling the same registry handlers. `build_workflow` (session tool) description gains "then dry_run_workflow, then run_workflow".

### P4 — The Sentinel (post-failure pickup, deterministic, zero-LLM)
- Extend `instincts.onRunFailed` seam: on terminal FAILED / COMPLETED_WITH_ERRORS of a **non-debug** run, file/refresh a deduped Issue (existing issues table/route) titled by failure signature, carrying `analyzeRunFailure` diagnosis + compass `nextCalls` (diagnose/replay/patch with real ids). Debug runs never file — the agent is already watching.

### P5 — UI: Build Health chip
- `WorkflowHealthIndicator` reads `settings.buildLoop` → stage chip (Draft → Authored → Dry-run ✓ → Debug ✓ → Production) with timestamps; red issue count from last dry_run.

### P6 — Tests + fence
- Unit: compass stages + hash invalidation + next-step generation; one-door gates (workflow.create rejects coupling-broken graph; patch at-rest rejects regression, honors confirmIntentChange); dry_run stamps loop state; loop_status; initialize.instructions present; sentinel files deduped issue.
- `pnpm -C apps/api exec tsc --noEmit` + targeted vitest after every wave; full engine/service sweep at the end.

**Ship order: P1 → P0 → P2 → P3 → P4 → P5 → P6 throughout** (P1 first because P0's new results already want the compass).

---

## 4. Non-goals / guardrails
- **No second simulator, no new runtime, no third engine** — every phase extends an existing seam (`createWorkflowFromDescription`, the registry, `CONTROL_TOOLS` late-binding, `instincts.onRunFailed`, `WorkflowHealthIndicator`). ([[feedback_no_duplication]])
- **No model-family branching.** Compass output is plain data usable by any model. ([[feedback_model_agnostic_tools]])
- **No hard-block friction on arming** — compass warns; the operator/agent decides. The one-door gates block only what the organs already define as corruption.
- **Never claim green from piped output** — capture exit codes; re-run typecheck/tests after every edit round. ([[feedback_clean_no_loose_ends]])

---

## 5. Acceptance (the operator's demand, made testable)
> An external MCP agent connects cold. From `initialize.instructions` + tool descriptions alone it: builds a complex workflow through the one door, is handed dry_run in the result, fixes reds it is handed by name, debug-runs via the compass, arms, and when a production run later fails an Issue appears carrying the exact next calls — **without the agent ever having read the docs.** A session-runtime agent inside a run can build → dry_run → run a sub-workflow without leaving its loop.
